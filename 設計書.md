# GWS ユーザー情報一括更新ツール 設計書

最終更新: 2026-06-06

---

## 1. 概要・目的

Google スプレッドシートにバインドした Google Apps Script（GAS）で、Google Workspace（GWS）の
ユーザーを **作成・更新・削除・停止・再開** を 1 シート上で **混合して一括実行** するツール。

- 操作対象は Google Directory API（Admin SDK）で更新可能なユーザー属性。
- 画面・項目はすべて日本語表記。
- 対象規模: 中小企業（50〜300 ユーザー）。

## 2. 要件

| # | 要件 | 対応方針 |
|---|------|----------|
| R1 | ユーザーの作成・更新・削除 | `操作` 列で行ごとに指定 |
| R2 | API で更新可能なユーザー属性を対象 | Directory API の更新可能フィールドをほぼ網羅 |
| R3 | 作成・更新・削除を混合し一括実行 | 単一メニュー「一括実行」で行順にディスパッチ |
| R4 | スプレッドシート上で操作 | バインド型 GAS、`ユーザー一覧` シートで操作 |
| R5 | 項目を日本語で表記 | 列名・メニュー・ダイアログを日本語化 |
| R6 | 実行時間制限 30 分（GWS） | 既定 27 分でチェックポイントし自動再開 |
| R7 | 一括処理の高速化 | 事前照会の削減・ログ抑制・バッチ書き込み |

操作種別は要件の 作成/更新/削除 に加え、**停止/再開**（アカウント無効化/再有効化）を提供する。

## 3. アーキテクチャ

### 3.1 ファイル構成

```
GWSユーザー情報一括更新ツール/
  .clasp.json              # clasp 設定（scriptId 空, rootDir: "src"）
  README.md                # 利用者向け手順
  設計書.md                # 本書
  src/
    appsscript.json        # V8 / Asia/Tokyo / AdminDirectory / OAuth スコープ
    config.gs              # 設定・バッチ状態（PropertiesService）
    sheetUtils.gs          # シート読み書き・行ステータスのバッチ更新
    apiClient.gs           # Directory API 呼び出し（リトライ/レート制限/分類）
    logger.gs              # ログシート出力
    fieldMap.gs            # 列 ⇔ Directory API 属性の双方向マッピング
    gwsUsers.gs            # ユーザー CRUD プリミティブ・別名同期
    bulkRunner.gs          # 混合一括処理・状態機械・トリガー連鎖
    ui.gs                  # メニュー・設定/確認/結果ダイアログ
    main.gs                # エントリポイント・初期化・インポート
```

GAS は全ファイルを単一のグローバルスコープに連結する。各ファイルは `import` を使わず、
グローバル関数・定数を相互参照する（`/* global ... */` で静的解析に明示）。

### 3.2 モジュール依存（呼び出し方向）

```
main.gs ──┬─> ui.gs ───────────────┐
          │                         ├─> sheetUtils.gs
          ├─> bulkRunner.gs ──┬─────┤
          │                   ├─> gwsUsers.gs ─> apiClient.gs ─> (AdminDirectory)
          │                   ├─> fieldMap.gs
          │                   └─> logger.gs
          └─> config.gs  ◀── 全モジュールが参照（設定・バッチ状態）
```

- `fieldMap.gs` が列定義（`USER_SHEET_HEADERS`）と属性マッピングの唯一の真実。
- `config.gs` の `BATCH_STATE`（PropertiesService）がジョブ状態の永続化先。

- `操作` 列にはデータ入力規則（プルダウン）を設定。先頭 4 列を固定表示。

### 4.2 「設定」シート

キー/バリュー/説明の 3 列。`loadConfigFromSheet()` で PropertiesService に取り込む。

| キー | 既定 | 説明 |
|------|------|------|
| `GWS_DOMAIN` | （必須） | Google Workspace ドメイン |
| `GWS_DEFAULT_OU` | `/` | デフォルト組織単位パス |
| `BATCH_SIZE` | `100` | チェックポイント間隔（件） |
| `TIME_LIMIT_MS` | `1620000` | 1 実行の上限（27 分） |
| `MAX_RETRIES` | `5` | API リトライ上限 |
| `RETRY_BASE_DELAY_MS` | `1000` | リトライ基準間隔（ミリ秒） |
| `CUSTOM_SCHEMA_NAME` | （空） | よみがな/内線用カスタムスキーマ名 |
| `CUSTOM_FIELD_LASTNAME_YOMI` / `..._FIRSTNAME_YOMI` / `..._EXTENSION` | （空） | カスタム各フィールド名 |
| `DEBUG_LOGGING` | `false` | 詳細ログ（各 API 呼び出しを記録） |

### 4.3 「ログ」シート

`タイムスタンプ / レベル / モジュール / 操作 / メッセージ / 詳細 / 行参照 / ジョブID`。
ジョブ開始・完了と失敗行のみ記録（成功はホットパスで記録しない）。

## 5. 属性マッピング（fieldMap.gs）

シート行 ⇔ Directory API リソースの双方向変換を集約。

- `USER_SHEET_HEADERS` — 列順の唯一の定義。
- `parseRow_(headers, row)` — 行 → 中間表現 `p`。各項目グループに `*Touched` フラグを持つ
  （ユーザーが値を入れたか）。空欄は「変更しない」を意味する。
- `toCreateResource(p)` — 作成用リソース。`primaryEmail`・`name`（姓名必須）・`orgUnitPath` を
  含め、`applyCommonFields_` で単純フィールド群を載せる。
- `toUpdateResource(p, current)` — 更新（patch）用リソース。空欄は載せない。
  - `name` / `organizations` は配列・ネストが丸ごと置換されるため、部分指定時は
    `current`（現在値）から欠けたサブ項目を補い、データ消失を防ぐ。
  - `needsCurrentForUpdate(p)` が true（氏名/組織を編集）の行のみ現在値を 1 回 GET。
- `userToRow(user)` — API リソース → 行（インポート時）。

ビルダー（空欄なら null）: `buildGender_` / `buildNotes_` / `buildLanguages_` /
`buildWebsites_` / `buildIms_` / `buildKeywords_` / `buildLocations_` /
`buildEmails_` / `buildCustomSchemas_`。

## 6. 操作仕様と混合一括処理

### 6.1 ディスパッチ（bulkRunner.gs `processRow_`）

| 操作 | API | 補足 |
|------|-----|------|
| 作成 | `Users.insert` | 姓・名・パスワード必須。既存は ⏭️スキップ（「already exists」を捕捉） |
| 更新 | `Users.patch` | 非空セルのみ反映。氏名/組織編集時のみ 1 回 GET してマージ |
| 削除 | `Users.remove` | 破壊的（復元期間 約20日）。確認で二段階 |
| 停止 | `Users.patch {suspended:true}` | アカウント無効化 |
| 再開 | `Users.patch {suspended:false}` | アカウント再有効化 |

別名（`別名` 列）が指定された行は、作成/更新後に `Users.Aliases` で差分同期。

### 6.2 一括実行フロー（runBulk）

```
runBulk()
  1. loadConfigFromSheet（設定の最新化）/ ドメイン未設定なら中止
  2. 実行中ジョブの有無を確認
  3. 対象行を抽出（操作が有効 かつ ステータス∈{空,⏳待機中,❌失敗}）
  4. 操作別件数を集計し確認ダイアログ（削除は強調・二段階）
  5. 対象行を ⏳待機中 に一括設定
  6. ジョブ状態を保存（jobId・カウンタ・開始時刻）→ executeBulk()
```

### 6.3 バッチ状態機械（executeBulk）と 30 分制限

冪等な「待機中スキャン」方式を採用。インデックス管理を持たず、毎回シートを 1 回読み、
`ステータス==⏳待機中` の行だけを残りとして処理する（完了行は ✅/❌/⏭️ になり対象から外れる）。

```
executeBulk()
  シートを1回読む → 待機中かつ有効操作の行 = remaining
  for each row in remaining:
     if 経過 > TIME_LIMIT_MS: 中断フラグ → break
     processRow_ で API 実行 → 結果を updates に蓄積
     if updates 件数 >= BATCH_SIZE:
        updateRowStatusBatch（まとめて書き込み）/ saveBatchState
  残り updates を flush / saveBatchState
  中断時: executionCount++ → ScriptApp.newTrigger("resumeBatch").after(1000) で自動再開
  完了時: status=completed → 結果ダイアログ → トリガー後始末
```

- 時間トリガー `resumeBatch`（main.gs）が中断ジョブを再開し `executeBulk` を継続。
- ✅成功行は再処理されない（冪等）。50〜300 名規模は通常 1 実行内で完了。

## 7. 高速化設計（一括処理前提）

| 施策 | 内容 |
|------|------|
| 作成の事前照会廃止 | `getUser` を省き、`insert` 直接実行→「既存」エラーで ⏭️スキップ（API 半減） |
| 成功ログの抑制 | ログシート追記はジョブ開始/完了＋失敗のみ。`DEBUG_LOGGING=true` 時のみ各呼び出しを記録 |
| ステータスのバッチ書き込み | `BATCH_SIZE` 件ごとに `updateRowStatusBatch`（既存値を読み対象行のみ上書き＝飛び地の行を保持） |
| シート読み取り 1 回/実行 | API 呼び出しの合間（毎行）に時間チェック、チャンク境界で flush |
| 更新の GET 最小化 | `patch` 使用。氏名/組織を編集する行のみ現在値を 1 回取得 |
| レート制御 | 約 2400 req/min の枠を CacheService で監視し控えめに待機 |

## 8. 更新セマンティクス・安全性

- **部分更新**: 更新は非空セルのみ反映（空セルでは項目を消さない）。
- **マージ対象**: `name`・`organizations` は現在値とマージ（一部編集で他項目を保持）。
- **置換対象**: `phones`/`addresses`/`emails`/`locations`/`externalIds`/`relations`/
  `websites`/`ims`/`languages`/`keywords` は入力列でその種別を置換
  （インポートで全列が埋まるため通常運用では消失しない）。
- **削除保護**: 確認ダイアログで削除件数を強調し、削除時は二段階確認＋復元期間警告。
- **キー固定**: 更新/削除/停止/再開は `メールアドレス` を userKey として使用。

## 9. エラーハンドリング（apiClient.gs）

- `gwsRequest(apiCall, name)` が指数バックオフ付きで API を実行。
- Advanced Service の例外はメッセージで分類（HTTP コード＋本文キーワード）:
  - クォータ/レート（`rate limit`/`quota`/429 等）→ 長めバックオフで再試行
  - 一時エラー（500/502/503/504/408、`backend error`/`timeout` 等）→ 再試行
  - それ以外 → `ApiError`（FATAL）として即時失敗
- 行単位で失敗を捕捉し ❌失敗 を記録、他行は継続。失敗内容はメッセージ列とログ・
  結果ダイアログ（先頭 50 件）に残す。

## 10. 認証 / OAuth スコープ（appsscript.json）

- Admin SDK は **AdminDirectory Advanced Service** として利用（UrlFetch 不使用）。
- スコープ:
  - `spreadsheets.currentonly` — バインドされたシートのみ
  - `admin.directory.user` — ユーザー CRUD
  - `admin.directory.user.alias` — 別名管理
  - `script.scriptapp` — バッチ継続トリガー
- カスタムスキーマ定義を読む場合は `admin.directory.userschema.readonly` を追加。

## 11. 制約・除外事項

- **実行時間**: 1 実行 30 分（GWS）。既定 27 分でチェックポイントし自動再開。
- **既定で非対応のフィールド**（複雑・ニッチ。必要時に追加可能）:
  `posixAccounts`（Unix アカウント）、`sshPublicKeys`（SSH 公開鍵）、
  `hashFunction`（事前ハッシュ済みパスワード用）。
- **よみがな・内線**は標準属性に無く、カスタムスキーマ設定時のみ反映。
- 別名は `Users.update/patch` では設定できず、`Users.Aliases` API で同期。
- `archived=TRUE` には Archived User ライセンスが必要。

## 12. テスト・検証

- **静的**: `npm run lint`（ESLint、`.js`/`.gs` 対象）でエラー 0。
- **マッピング**: `fieldMap.gs` の `runFieldMapSelfTest()` を GAS エディタで実行し、
  行→中間表現→作成リソースの往復一致を確認（`OK` をログ出力）。
- **手動 E2E**（テスト用 GWS ドメイン・バインド済みシート）:
  1. `clasp push` 後、メニュー「シート初期化」「初期設定」（ドメイン）。
  2. 「GWSからインポート」で既存ユーザーを取り込み。
  3. `操作` に 作成/更新/削除/停止/再開 を混合設定 →「一括実行」。
  4. 確認ダイアログ → 各行のステータス/メッセージ・Admin コンソール・ログを照合。
  5. 大量データでトリガー連鎖（30 分超分割）と再開・キャンセル、異常系（権限/重複/
     レート制限）で他行継続を検証。
