# Google グループ一括管理ツール 設計書(2/2)処理方式設計

**版数:** v0.1(レビュー用) **作成日:** 2026-06-12 **対応文書:** 要件定義書 v1.0、設計書(1/2)シート設計 v1.0

## 1. 本書の位置づけ

設計フェーズの第 2 分冊として、処理方式(ジョブ実行・継続、予約・定期トリガー、API 対応、種別判別、エラー処理、CSV 出力、権限)を定める。

## 2. 利用 API と事前準備

### 2.1 拡張サービス・API

| サービス | 用途 | 有効化 |
|----------|------|--------|
| Admin SDK Directory API | グループ・メンバーの CRUD、一覧取得 | GAS 拡張サービス + 管理コンソールで API アクセス許可 |
| Groups Settings API | 設定 27 項目の取得・変更 | GAS 拡張サービス |
| Cloud Identity Groups API | 動的・セキュリティグループの判別(読み取りのみ) | GAS 拡張サービス(識別子は実装時に manifest で確定) |

### 2.2 API 対応表

| 操作 | API / メソッド | 備考 |
|------|----------------|------|
| グループ作成 | Directory: Groups.insert | email・name・description |
| グループ削除 | Directory: Groups.remove | 削除確認列が「削除する」の行のみ |
| 基本情報変更(email / name / description) | Directory: Groups.patch | 指示_設定変更の B〜D 列 |
| 設定変更(E〜AB 列) | Groups Settings: Groups.patch | 記入セルのみをボディに含める |
| メンバー追加 | Directory: Members.insert | role 指定(既定 MEMBER) |
| メンバー削除 | Directory: Members.remove | |
| 所属確認(冪等判定) | Directory: Members.get | 404 = 未所属 |
| グループ一覧 | Directory: Groups.list | customer='my_customer'、maxResults=200、pageToken |
| メンバー一覧 | Directory: Members.list | グループごとにページネーション。直接メンバーのみ取得 |
| 設定一覧(棚卸_設定) | Groups Settings: Groups.get | グループごとに 1 回 |
| 種別情報の一括取得 | Cloud Identity: groups.list(view=FULL) | labels・dynamicGroupMetadata を取得 |
| 種別情報の個別取得 | Cloud Identity: groups.lookup → groups.get | 実行時バリデーション用 |

※ 指示_設定変更の 1 行に B〜D 列(Directory 側)と E〜AB 列(Groups Settings 側)が混在する場合は、Directory → Groups Settings の順に 2 回の API 呼び出しで適用する。email 変更を含む場合は、変更後アドレスで Groups Settings を呼び出す。

### 2.3 設定 27 項目の列挙値(マスタ_選択肢の初期値)

各「選択」項目の列挙値は Groups Settings API リファレンスに準拠し、初期セットアップ機能がマスタ_選択肢シートへ自動投入する。値の一覧はコーディング時に API リファレンスから転記して確定する(本書では方式のみ定義)。

## 3. 動的・セキュリティグループの判別方式

| 区分 | 判別条件 |
|------|----------|
| セキュリティ | Cloud Identity の labels に `cloudidentity.googleapis.com/groups.security` を含む |
| 動的 | dynamicGroupMetadata を持つ(または labels に `cloudidentity.googleapis.com/groups.dynamic` を含む) |
| 通常 | 上記いずれにも該当しない |

- **棚卸し時:** Directory: Groups.list(基本情報)と Cloud Identity: groups.list(種別情報)をそれぞれページネーションで全件取得し、グループメール(groupKey.id)で突合して種別列を埋める(個別呼び出しを避けて効率化)。
- **実行時(FR-09):** 変更・削除系操作の対象グループごとに lookup → get で判別し、ジョブ実行中はメモリ上にキャッシュして重複呼び出しを避ける。動的・セキュリティ該当時は結果「対象外」とし API を呼び出さない。

## 4. ジョブ実行・継続方式(FR-16, NFR-02)

### 4.1 ジョブ状態管理

- ジョブ状態は ScriptProperties の `JOB_STATE` に JSON で保持する:実行 ID、機能、モード、対象シート、行ポインタ(指示実行)または pageToken・処理済みグループ位置(棚卸し)、集計値(成功・失敗・スキップ)、キャンセルフラグ。
- 指示実行の処理対象は「結果列が 空欄 / 検証OK / 検証NG / 失敗 の行」であるため、結果列自体が永続的な進捗記録を兼ねる(JOB_STATE 喪失時も安全に再開可能)。

### 4.2 実行フロー

1. **開始:** LockService でスクリプトロックを取得。`JOB_STATE` が存在する場合は多重実行としてエラー表示(継続トリガー起動時を除く)。実行 ID を発行しログに「開始」を記録。
2. **チャンク処理:** 対象行を上から順に処理し、BATCH_ROWS 行ごとに結果 4 列を書き戻して `JOB_STATE` を更新する。
3. **時間監視:** 行処理の都度、経過時間を確認。TIME_LIMIT_SEC(既定 300 秒)を超えたら、書き戻し → 継続トリガー(約 1 分後に起動するワンタイム)を作成 → ログ状態を「継続中」に更新して終了。
4. **継続起動:** 継続トリガーのハンドラは、まず自トリガーを削除 → `JOB_STATE` を復元 → モード「継続」でステップ 2 から再開。
5. **完了:** 全行処理後、集計をログへ記録(状態「完了」)、`JOB_STATE` と残トリガーを削除。棚卸しジョブは完了時に CSV を自動出力する(D-2)。
6. **中断:** メニュー「実行中ジョブの中断」は `JOB_STATE` にキャンセルフラグを立て、次のチャンク境界で書き戻しのうえ状態「中断」で停止する(処理済み行はそのまま有効。再実行で残りから継続)。

### 4.3 棚卸しの分割

- グループ一覧:Groups.list の pageToken を `JOB_STATE` に保持して継続。
- メンバー一覧:棚卸_グループの行を順に処理し、「処理中のグループ位置 + Members.list の pageToken」を保持して継続。
- シートへの書き込みは配列バッファに貯めて一括 setValues(数百行単位)で行い、実行時間を節約する。

## 5. 予約・定期実行方式(FR-14, FR-15)

### 5.1 予約実行(指定日時のピンポイント実行)

- メニュー「予約実行を登録」:設定シートの RESERVE_DATETIME・RESERVE_TARGET を読み込み、過去日時はエラー。既存の予約トリガーがあれば削除のうえ、指定日時のワンタイムトリガーを作成し、トリガー ID を ScriptProperties に保持する。
- 予約ハンドラ:起動直後に自トリガーを削除(FR-14)→ RESERVE_TARGET に応じて本実行ジョブを 4 章の機構で起動する。予約はドライラン不可(本実行のみ)。ドライランは登録前に手動で実施する運用とする。
- 「登録状況を確認」で予約日時・対象を UI 表示。「解除」でトリガー削除。

### 5.2 定期棚卸し

- INVENTORY_SCHEDULE(「毎日 HH:mm」または「毎週 曜 HH:mm」)を解析し、時間主導型トリガーを作成する。INVENTORY_MEMBERS=TRUE の場合はグループ → メンバーの順に連続実行する。棚卸_設定は定期対象外(手動メニューのみ。D-1)。
- 注意:GAS の時間主導型トリガーは指定時刻から前後 15 分程度ずれることがある(仕様)。分単位の厳密性が必要な場合は予約実行(at 指定)を用いる。

### 5.3 トリガー本数の管理

本ツールが作成するトリガーは最大 3 本(予約 1・定期棚卸し 1・継続 1)とし、上限(1 ユーザー・1 スクリプトあたり 20 本)に対して十分な余裕を確保する。作成したトリガー ID はすべて ScriptProperties で管理し、「登録状況を確認」で一覧表示、完了・解除時に確実に削除する。

## 6. バリデーション仕様(FR-09)

| 対象 | チェック内容(検証NG とする条件) |
|------|----------------------------------|
| 共通 | メール形式不正、必須項目の欠落、操作種別・ロール・列挙値が値域外 |
| グループ作成 | (既存と重複 → API 409 で「スキップ」扱い) |
| グループ削除 | 削除確認が「削除する」以外、対象が存在しない(→「スキップ」) |
| メンバー追加/削除 | 対象グループが存在しない。追加済み/未所属は「スキップ」 |
| 設定変更 | 記入セルの値が列挙値・型(TRUE/FALSE)に合わない、対象グループが存在しない |
| 変更・削除系すべて | 対象が動的・セキュリティグループ →「対象外」(API を呼ばない) |

- ドライランは上記すべて+存在確認系の読み取り API のみ実行し、「検証OK / 検証NG」を記録する。
- 検証 NG はその行のみ記録して処理を継続する(他行を止めない)。

## 7. エラー処理・リトライ(NFR-03)

| 事象 | 処理 |
|------|------|
| 429 / rateLimitExceeded / quotaExceeded / 5xx | 指数バックオフでリトライ:待機 = RETRY_BASE_MS × 2^試行回数 + ランダムジッタ、最大 RETRY_MAX 回。超過時は「失敗」 |
| 404(対象なし) | 操作に応じて「スキップ」(削除系)または「検証NG」(対象必須の操作) |
| 409(重複) | 作成・追加では「スキップ」(冪等) |
| その他の API エラー | 「失敗」とし、メッセージ列に API エラー概要を記録。処理は次行へ継続 |
| スクリプト例外(想定外) | ログ状態「エラー」で記録し、JOB_STATE は保持(再実行で継続可能) |

API 呼び出しは共通ラッパ(ApiClient)経由とし、リトライ・待機・エラー整形を一元化する。

## 8. CSV 出力仕様(FR-08, D-2)

| 項目 | 内容 |
|------|------|
| 出力先 | 設定シート CSV_FOLDER_ID のドライブフォルダ |
| ファイル名 | `棚卸_グループ_yyyyMMdd-HHmmss.csv` / `棚卸_メンバー_…` / `棚卸_設定_…` / `実行結果_<指示シート名>_<実行ID>.csv` |
| 文字コード | UTF-8(BOM 付き。Excel での文字化け防止) |
| 形式 | カンマ区切り、1 行目に見出し。カンマ・改行・引用符を含む値はダブルクォートで囲む |
| 契機 | 棚卸し完了時に自動出力、およびメニュー「CSVへ出力」による手動出力(指示シートの実行結果を含む) |

## 9. OAuth スコープ(NFR-07)

manifest(appsscript.json)の oauthScopes に以下を明示宣言し、最小化する。

| スコープ | 用途 |
|----------|------|
| …/auth/admin.directory.group | グループ・メンバーの読み書き |
| …/auth/apps.groups.settings | 設定 27 項目の取得・変更 |
| …/auth/cloud-identity.groups.readonly | 種別判別(読み取りのみ) |
| …/auth/spreadsheets.currentonly | 本スプレッドシートの読み書き |
| …/auth/drive | CSV の指定フォルダへの出力(drive.file では既存フォルダへ出力できないため) |
| …/auth/script.scriptapp | トリガーの作成・削除 |
| …/auth/userinfo.email | 実行者の記録(ログ) |

## 11. コーディングの進め方(実装順)

1. 03_Setup / 01_Config / 13_Logger / 14_ApiClient(共通基盤)
2. 04_BatchRunner(継続実行エンジン)+ ダミー処理での動作確認
3. 09_Inventory + 10_GroupTypeResolver(読み取り系から着手し安全に検証)
4. 05_Validators + ドライラン
5. 06〜08 の変更系(テスト用グループで検証)
6. 11_TriggerManager(予約・定期)、12_CsvExporter
7. テストフェーズへ(テスト仕様書はテスト用グループでの検証を前提に別途作成)

## 12. 残課題

- マスタ_選択肢の初期値(各列挙値の一覧)は、コーディング時に Groups Settings API リファレンスから転記して確定する。
- Cloud Identity 拡張サービスの識別子・有効化手順は実装時に manifest で確定する。
