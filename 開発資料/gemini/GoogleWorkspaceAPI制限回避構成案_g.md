# Google Workspace API制限回避の推奨構成案

Google Workspace Admin SDK（ユーザー/グループ更新）において、APIの制限（クォータ）を回避し、数千〜数万件規模の更新を安定して実行するための構成案です。

## 1. 構成の全体像
以下の3つの手法を組み合わせ、**「差分抽出 → タスク分割 → 流量制御付き実行」**のパイプラインを構築します。

1.  **差分同期 (Differential Sync)**: 変更があったユーザーのみを特定。
2.  **Cloud Tasks による処理の断片化**: 大量更新を小さな単位（チャンク）に分割。
3.  **バッチリクエストと流量制御 (Throttling)**: APIリクエストを集約し、一定のペースで実行。

---

## 2. 各手法の詳細

### 手法①：差分同期（Differential Sync）
全件更新ではなく、前回の同期状態と今回の入力データを比較し、実際に変更（追加・更新・削除）が必要なデータのみを抽出します。

- **実装方法**:
    - Cloud SQLに「前回同期成功時のデータハッシュ値」を保持。
    - 入力データ（CSV等）とハッシュ値を比較し、不一致のレコードのみを抽出。
- **効果**: API呼び出し回数を、全ユーザー数から「実際の異動者数」まで劇的に削減（通常、全件の1%〜5%程度）。

### 手法②：Cloud Tasks による処理の分散
一度に大量のAPIリクエストを投げず、Cloud Tasks（キュー）を利用して時間軸で分散実行します。

- **実装方法**:
    - Cloud Runが同期リクエストを受信。
    - 対象ユーザーを100名単位の「チャンク」に分割。
    - 各チャンクをCloud Tasksへタスクとして投入。
- **効果**: 
    - APIの「100秒あたりのリクエスト制限」を超えないよう、Cloud Tasksのディスパッチ速度（Dispatch Rate）を調整可能。
    - APIエラー時にCloud Tasksが自動リトライ（指数関数的バックオフ）を管理。

### 手法③：バッチリクエストとウェイト制御（Throttling）
Google APIの「Batch Request」機能を使い、さらにプログラム側で明示的な待機時間を設けます。

- **実装方法**:
    - 1つのHTTPリクエストに最大100件のAPI呼び出しを集約。
    - 10バッチ（1,000件分）送信するごとに、数秒間のウェイト（time.sleep）を挿入。
- **効果**: 
    - HTTP通信のオーバーヘッドを削減。
    - 秒間リクエスト制限（QPS）の超過を物理的に防止。

---

## 3. 処理フローのまとめ

1.  **管理操作**: AppSheetで「同期実行」ボタンを押下。
2.  **差分抽出**: Cloud RunがDBと入力を比較し、更新が必要な1,500名を特定。
3.  **タスク分割**: 15個のタスク（100名×15）をCloud Tasksへ投入。
4.  **分散実行**: Cloud Tasksが、設定された秒間レートに従って順次Cloud Run（Worker）を起動。
5.  **API発行**: Workerが100名分のデータを、バッチリクエストとしてGoogle Workspace APIへ送信。
6.  **結果記録**: 処理完了後、DBのハッシュ値と実行ログを更新。

---

## 4. 期待される効果
- **API制限（403 User Rate Limit Exceeded）の完全回避。**
- **大規模メンテ時の安定性向上**: 数万件の異動期でもシステムがダウンせず、着実に更新を完了。
- **運用の透明性**: Cloud Tasksのコンソールから、処理の進捗状況や残りのタスク数、エラー率をリアルタイムに監視可能。
