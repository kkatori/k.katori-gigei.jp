# 実現性評価レポート：ユーザー管理機能

## 1. 評価総括
**実現性：高（Feasible）**

提示された要件は、Google Cloud (Cloud Run, Cloud SQL) と AppSheet を中核としたアーキテクチャにより、技術的に十分に実現可能です。
ただし、「ローカルファイルサーバーへの自動保存」に関しては、クラウド環境から社内ローカルネットワークへの直接アクセスとなるため、セキュリティおよびネットワーク構成上の制約があります。これについては代替案（クライアントPC経由での実行など）による解決を推奨します。

---

## 2. コンポーネント別 実現性検証

### 2.1 フロントエンド (AppSheet)
*   **要件**: ユーザー・グループの一覧表示、入力、CSVアップロード、ログ表示。
*   **評価**: **実現可能**
*   **詳細**:
    *   **DB連携**: Cloud SQLをAppSheetのデータソースとして設定することで、大規模なユーザーデータもスムーズに扱えます。
    *   **CSVアップロード**: AppSheetの標準機能（ファイル型カラム）でCloud Storageへファイルをアップロード可能です。
    *   **処理実行**: Automation機能（Webhook）を利用することで、ボタン操作をトリガーにしてCloud Run上のバッチ処理を呼び出せます。
    *   **制約事項**: AppSheetは画面デザイン（UI）の自由度が低いため、「ピクセル単位のレイアウト指定」などはできませんが、管理画面としての機能要件は満たせます。

### 2.2 バックエンド (Google Cloud)
*   **要件**: Cloud Run/App Engineでのバッチ処理、Cloud SQLでのデータ管理。
*   **評価**: **実現可能**
*   **詳細**:
    *   **スケーラビリティ**: サーバーレス構成（Cloud Run）のため、バッチ処理時のみリソースを消費する効率的な運用が可能です。
    *   **認証**: Google Cloud IAMおよびIdentity-Aware Proxy (IAP) を利用することで、Google Workspaceアカウントを用いた強固なセキュリティ（SSO）を実装できます。

### 2.3 外部システム連携 (API)

#### A. Google Workspace 連携
*   **要件**: ユーザー・グループのCRUD操作（作成・参照・更新・削除）。
*   **評価**: **実現可能**
*   **手段**: Admin SDK Directory API を使用します。
*   **注意点**: APIには利用制限（クォータ）があるため、数千件単位の一括更新を行う場合は、バッチ処理側でAPIコールの流量制御（スロットリング）と再試行ロジックの実装が必須です。

#### B. rakumo 連携
*   **要件**: rakumoコンタクトの更新、同期。
*   **評価**: **実現可能**
*   **手段**: rakumoが提供するAPI（Google Sync API, CSV Upload API等）を利用します。
*   **確認事項**: 資料に記載のAPI仕様（`https://sites.google.com/gigei.jp/...`）に基づき、認証方式とCSVフォーマットの事前検証が必要です。

### 2.4 ローカルシステム連携 (CSVダウンロード)
*   **要件**: 生成したCSVをファイルサーバー（ローカルネットワーク）へ保存。
*   **評価**: **条件付きで実現可能**（構成上の課題あり）
*   **課題**: Cloud Run（パブリッククラウド）から、社内LAN内のファイルサーバーへ直接ファイルを書き込むことは、通常セキュリティ上遮断されています。
*   **解決策（推奨案）**:
    1.  **クライアント実行（要件定義書の記載に近い）**: 
        ユーザーがAppSheetまたはWeb画面から「CSVダウンロード」ボタンを押し、手元のPCに保存した後、手動またはPC上のスクリプトでファイルサーバーへ移動する。
    2.  **専用エージェントの設置**:
        社内サーバー上に小さなプログラムを常駐させ、定期的にCloud Storageを見に行き、新しいCSVがあればダウンロードしてファイルサーバーに保存する（ポーリング方式）。
    3.  **VPN/Interconnect**:
        Cloud VPN等でGCPと社内NWを接続する（コストと構築期間がかかるため、本件の規模感ではオーバースペックの可能性あり）。

## 3. 技術的リスクと対策

| リスク項目 | リスク内容 | 対策・対応方針 |
| :--- | :--- | :--- |
| **AppSheetの行数制限** | AppSheetは読み込み行数が多いと動作が重くなる（通常数万行レベル）。 | 表示用データはCloud SQL側でビューを作成して必要な分だけフィルタリングするか、AppSheetの「Security Filter」機能を活用する。 |
| **APIレート制限** | Google Workspace API等の制限を超え、更新が失敗する。 | バックエンド処理（Cloud Run）にて、エクスポネンシャルバックオフ（指数関数的リトライ）とレートリミット制御を実装する。 |
| **データ整合性** | API更新中にエラーが発生し、DBと実態（Workspace）がズレる。 | 処理結果を必ずDBのログテーブルに記録し、失敗分のみ再実行できる「リトライ機能」または「差分同期機能」をUIに設ける。 |

## 4. 結論
提示された要件定義書の内容は、**技術的に妥当であり実現可能**です。
特に「ローカルファイルサーバー連携」の方式についてのみ、開発フェーズの初期段階で具体的な運用フロー（「人の手でダウンロードする」か「自動化プログラムを社内に置く」か）を確定させることを推奨します。
