### [冪等-1] GWSエラー時のDB・GWS状態乖離後の再実行フローが未定義 ★重大

**該当箇所：** 10.3節 POST `/api/users/import` 処理順序・GWSエラー時の記述

**問題内容：**
処理順序が以下のように定義されている。
```
①CSVバリデーション → ②Cloud SQL更新（トランザクション） → ③GWS Admin SDK呼び出し → ④ログ記録
```
「GWSエラー時はCloud SQL更新はロールバックせず、エラーログを記録して再実行可能とする」と記載があるが、**再実行時の具体的な動作フローが未定義**。

再実行時に問題が生じるパターン：

| シナリオ | Cloud SQL状態 | GWS状態 | 再実行結果 |
|---------|-------------|---------|----------|
| ②成功→③失敗後に再実行 | 更新済み | 未反映 | Cloud SQLを再度UPSERTしてGWSにも反映 → 問題なし（設計による）|
| ②成功→③一部成功一部失敗後に再実行 | 更新済み | 部分反映 | GWSで既成功分に対して再度API呼び出し → 既存データへのUPSERT（GWS側の挙動確認が必要）|
| ①～③完了後に同一CSVを再実行 | 最新状態 | 最新状態 | Cloud SQLはUPSERTで問題なし。GWS既存ユーザーへのupdateは冪等 ✓ |

**対応方針（案）：**
- 再実行時はCloud SQL・GWSともに UPSERT（冪等）として扱う設計を10.3に明記する
- 処理順序を「②Cloud SQL UPSERT → ③GWS UPSERT（既存は update、新規は insert）」と明示する
- 部分失敗（GWSエラー発生行のみ）をログに記録し、該当行のみ再実行できる仕組みを検討する

### [冪等-2] Cloud Storageのファイル命名規則が再実行時に重複ファイルを生成する ★中

**該当箇所：** 6.3節（ファイル命名規則）

**問題内容：**
ファイルパスにタイムスタンプ（`{YYYYMMDD_HHMMSS}`）を含む命名規則のため、同一CSVを再実行すると異なるパスにファイルが重複保存される。

```
1回目： users/20260308_060000_users.csv
再実行： users/20260308_060015_users.csv  ← 別ファイルとして残存
```

これ自体は機能的に問題はないが、**90日間ファイルが残存し続ける**ため、誤ったCSVが残っていた場合に混乱が生じる可能性がある。

**対応方針（案）：**
- タイムスタンプ命名は意図的な設計であれば「再実行は別ファイルとして保存する（上書きしない）」と明記する
- または、処理成功時は処理済みフォルダ（`processed/`）に移動し、失敗時は失敗フォルダ（`failed/`）に移動する命名・管理方針を追記する

### [整合-2] `/api/google-sync` 処理順序のポーリング方法が要件定義書v1.4と不一致 ★中

**該当箇所：** 10.3節 POST `/api/google-sync` 処理順序

**問題内容：**
設計書には「③selfLink でポーリング」と記載されているが、要件定義書v1.4（4.4節・12.5節）では「`GET /api/google-sync/v1/jobs/current` を30秒間隔でポーリング」に修正されている。

```
現在の設計書：③selfLink を GET してポーリング
要件定義書v1.4：③GET /api/google-sync/v1/jobs/current を30秒間隔でポーリング（最大10分）
```

**対応方針（案）：**
10.3節の処理順序を以下に修正する。
```
①GETCURRENT で status 確認 → ②waiting であれば CREATE 実行 →
③GET /jobs/current を30秒間隔でポーリング（最大10分、status が waiting/success になるまで）→
④GETLASTRESULT で結果取得 → ⑤sync_logsに記録
```

### [整合-3] 3.2のデータフローにGoogle Sheets同期フローが記載されていない ★中

**該当箇所：** 3.2節（データフロー）

**問題内容：**

```
Cloud Run → Google Sheets API（AppSheet用スプレッドシートの同期）
```

**対応方針（案）：**

### [整合-4] Cloud Schedulerタイムアウトの設定説明が誤解を招く ★中

**該当箇所：** 9.2節（スケジューラ設定）

**問題内容：**
「タイムアウト：最大3,600秒（Cloud Schedulerのデフォルトは3分のため要設定）」と記載されているが、これは不正確。

| 項目 | 実際の制限 |
|------|----------|
| Cloud Schedulerのリクエストタイムアウト（deadline） | デフォルト3分、**最大30分（1,800秒）** |
| Cloud Runのリクエストタイムアウト | 最大3,600秒 |

Cloud Schedulerは最大30分のdeadlineを設定できるが、**Cloud Runの3,600秒まで待機できない**。Google同期や大量バッチ処理が30分を超える場合、Cloud Schedulerは処理中にタイムアウトしてエラーとなる（Cloud Runは処理を継続する）。

**対応方針（案）：**
以下のいずれかの設計を選択・明記する。
- (A) 処理を即時レスポンス（202 Accepted）で返し、バックグラウンド処理として続行する（非同期設計）
- (B) Cloud Run Jobs を使用してバッチ処理を実行し、Cloud SchedulerからCloud Run Jobsを起動する（Jobsはタイムアウト制限がない）
- (C) 処理が30分以内に完了することを設計上保証し、deadlineを1,800秒に設定する

## 3. 実現可能性の観点

### [実現-1] 長時間バッチ処理でCloud Schedulerタイムアウトが先に切れる ★重大

**該当箇所：** 9.2節（スケジューラからCloud Runへの呼び出し）

**問題内容：**
数万件のGWS登録・更新処理、Google同期完了待ち（最大10分ポーリング）、rakumoアップロード処理を組み合わせると、処理完了まで30分を超える可能性がある。
Cloud SchedulerのHTTPリクエストタイムアウトは最大30分（1,800秒）のため、**Cloud Runが処理中にCloud Schedulerが先にタイムアウト**し、Schedulerはエラーを記録する（Cloud Runの処理は継続するが、Schedulerからは成功可否が不明）。

**影響：**
- Cloud Schedulerのログが常にエラーになり、アラートが誤発報される
- 処理完了の通知が取得できない

**対応方針（推奨）：**
Cloud Run Jobs を使用してバッチ処理を実行する設計に変更する。

```
変更前：Cloud Scheduler → Cloud Run Services (HTTP)
変更後：Cloud Scheduler → Cloud Run Jobs (ジョブ実行)
```

Cloud Run Jobsはタスクの完了まで待機し、タイムアウト制限が3,600秒（最大86,400秒=24時間まで設定可能）のため、長時間処理に適している。または、Cloud Run Servicesを維持する場合は非同期レスポンス（202 Accepted）で即時返却し、処理はバックグラウンドで継続する設計とする。

### [実現-2] rakumoコンタクト更新APIの処理順序に完了確認ステップが含まれていない ★中

**該当箇所：** 10.3節 POST `/api/rakumo/contacts/update` 処理順序

**問題内容：**
処理順序が以下となっているが、要件定義書v1.4（4.5節）で規定した完了確認（LOCKポーリング）が含まれていない。

```
現在の設計書：
①Cloud SQLデータ取得 → ②CSV生成 → ③Cloud Storage保存 →
④IMPORT LOCK確認 → ⑤CREATE URL → ⑥UPLOAD → ⑦sync_logsに記録
```

アップロード（⑥）は非同期処理（202 Acceptedで即返却）のため、⑦でsync_logsに記録した時点では実際の処理は完了していない。

**対応方針（案）：**
処理順序に完了確認ステップを追加する。
```
①Cloud SQLデータ取得 → ②CSV生成 → ③Cloud Storage保存 →
④IMPORT LOCK確認（ロック中なら待機） → ⑤CREATE URL →
⑥UPLOAD（202 Accepted）→
⑦IMPORT LOCK をポーリング（ロック解除をもって完了判断）→
⑧sync_logsに記録
```

### [網羅-1] 4.6スプレッドシート作成機能のAPIエンドポイントが存在しない ★重大

**該当箇所：** 10.1節（エンドポイント一覧）

**問題内容：**

AppSheetがGoogle Sheetsをデータソースとして参照する設計であるため、このエンドポイントは必須。

**不足エンドポイント（案）：**

| メソッド | パス | 機能 | 呼び出しタイミング |
|---------|------|------|-----------------|
| POST | `/api/sheets/sync/users` | ユーザー情報Sheetsを同期 | ユーザーインポート後 |
| POST | `/api/sheets/sync/groups` | グループ情報Sheetsを同期 | グループインポート後 |
| POST | `/api/sheets/sync/logs` | ログSheets一括同期 | バッチ処理後 |
| POST | `/api/sheets/sync/all` | 全Sheetsを一括同期 | 定期バッチ後 |

また、8.4節（Google Workspace Admin SDK権限）にGoogle Sheets APIのスコープが記載されていない。

```
# 追加が必要なスコープ
https://www.googleapis.com/auth/spreadsheets
```

**対応方針（案）：**
- 10.1のエンドポイント一覧にSheets同期エンドポイントを追加する
- 8.4のサービスアカウントスコープにSpreadsheets書き込みスコープを追加する

### [網羅-2] `/api/csv/download` が署名付きURL発行エンドポイントであることが不明確 ★中

**該当箇所：** 10.1節 GET `/api/csv/download`

**問題内容：**

```
・レスポンス：{"download_url": "https://storage.googleapis.com/...", "expires_at": "..."}
```

### [網羅-3] 要件定義書v1.4で追加された初期データ移行・AppSheetアクセス制御の設計がない ★中

**該当箇所：** 全体（対応セクションなし）

**問題内容：**
要件定義書v1.4で新設された以下の要件に対応する設計が設計書に存在しない。

| 要件 | 要件定義書箇所 | 設計書での対応 |
|------|------------|-------------|
| AppSheetのドメイン制限 | 10.5節 | 記載なし |
| 初期データ移行方針 | 10.6節 | 記載なし |
| エラー通知先・通知条件 | 7章 非機能要件 | 7.2節で部分的に記載あり（通知先は「お客様指定」で具体性に欠く）|

**対応方針（案）：**
- AppSheetのドメイン制限は AppSheet設計書の範囲だが、Cloud Run側のアクセス制御（OIDCトークンのユーザードメイン検証）についてはCloud Run設計書の8章（セキュリティ設計）に追記する
- 初期データ移行についてはCloud Run設計書への記載は不要（AppSheet+CSVインポートで対応）だが、Cloud SQLの初期スキーマ適用手順（Alembicマイグレーション実行コマンド）を11章（環境設計）に追記することを推奨する
