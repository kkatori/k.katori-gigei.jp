## 観点1：冪等性

### 【重大】R1-01：CSVインポート重複実行時のGWS API多重呼び出し

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.1、Google Cloud設計書 10.3 |
| 問題 | 同一CSVを複数回インポートした場合、Cloud SQL へのUPSERT は冪等だが、GWS Admin SDK API が都度呼び出される設計になっている。差分なしでも GWS API が実行されてしまう |
| 対応方針 | GWS API 呼び出し前に「Cloud SQL 更新前後でデータに変化があった場合のみ API を呼び出す」差分チェックロジックを追加する。または UPSERT で変化がなければ API 呼び出しをスキップする方針を明記する |

### 【重大】R1-02：Google同期・rakumoコンタクト更新の409エラー後のリトライ仕様が不明確

| 項目 | 内容 |
|------|------|
| 対象ファイル | Google Cloud設計書 10.3、AppSheet設計書 7.2・7.3 |
| 問題 | Cloud Run が実行中（running/locked）の場合 409 エラーを返却するが、AppSheet 側でのリトライ方法が記載されていない。ユーザーが手動で再実行する必要があることが不明 |
| 対応方針 | AppSheet設計書のアクション詳細（A-07・A-08）に「409エラー時のメッセージ表示仕様（"処理が実行中です。しばらく後に再実行してください"等）」を追記する |

### 【中程度】R1-03：スプレッドシート更新中断時のデータ不整合

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.5、AppSheetスプレッドシート設計書 3 |
| 問題 | `values.clear → values.update` の全件書き込み方式で、clear 実行後に update が失敗した場合、シートが空になる。Cloud SQL は正常だがスプレッドシートが空という状態が発生しうる |
| 対応方針 | clear → update の処理を同一 Sheets API `batchUpdate` トランザクション内で実施するか、clear を使わず update のみとする実装方針を明記する |

## 観点2：整合性

### 【重大】R2-01：ユーザーマスタのカラム数が設計書間で不一致

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 5.1、Google Cloud設計書 5.4.1、AppSheetスプレッドシート設計書 2.1 |
| 問題 | 要件定義書は「ユーザーマスタ47項目」と明記しているが、Cloud SQL DDL には email・family_name 等の基本17項目 + rakumo制御フラグ8項目 + 組織情報22項目 = 47カラムに加え、status・created_at・updated_at・deleted_at の4カラムが存在し合計51カラムとなる。一方、スプレッドシート設計書は「49列（47項目＋status・updated_at）」と記載している |
| 対応方針 | 「ユーザーマスタ47項目」は業務データを指し、DBシステムカラム（status・created_at・updated_at・deleted_at）は別扱いであることを各設計書に明記する。スプレッドシート設計書には「49列＝業務47項目＋status・updated_at（deleted_at・created_atは非表示）」と補足を追加する |

### ~~【重大】R2-02：sync_logsテーブルとAppSheetテーブルの対応が不明確~~ ✅ 解決済み

| 項目 | 内容 |
|------|------|
| 対象ファイル | Google Cloud設計書 5.4.5、AppSheet設計書 2.5・2.6 |
| ~~問題~~ | ~~Cloud SQL の `sync_logs` テーブルは `sync_type ENUM('google_sync','rakumo_contact')` で1テーブルに集約されているが、AppSheet側はGoogleSyncLogs・RakumoContactLogs の2テーブル（2スプレッドシート）として分離されている。Cloud RunがどちらのシートにどのようにDB内容を書き出すかが不明確~~ |
| 対応済み | AppSheet設計書・AppSheetスプレッドシート設計書の `GoogleSyncLogs` / `RakumoContactLogs` を `SyncLogs`（1シート）に統合。`sync_type` カラムを追加し、AppSheetのビューでフィルタして表示を分割する方針に修正した |

### 【重大】R2-03：グループ種別フィルタのmailing表示と要件の不整合

| 項目 | 内容 |
|------|------|
| 対象ファイル | AppSheet設計書 3.2（グループ管理画面）、要件定義書 8.1 |
| 問題 | AppSheet設計書のグループ種別フィルタに「メーリングリスト用（mailing）」が選択肢として存在するが、要件定義書 8.1 ではメーリングリストは例示扱いで初期更新対象外。メーリングリストのデータがDBに存在しない場合、フィルタが空一覧を返す可能性がある |
| 対応方針 | AppSheet設計書のグループ種別フィルタに「初期運用ではdrive・rakumoのみデータが存在する。mailingは将来拡張用」と注記を追加する |

### 【中程度】R2-04：プロファイルCSV生成時のカラム対応が不明確

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.4、要件定義書 5.1・11.5 |
| 問題 | プログラム設計書の `RAKUMO_PROFILE_COLUMNS`（28列）と、ユーザーマスタ（47項目）の対応関係（どのDBカラムがどのCSV列に対応するか）が明示されていない。例：`email` → User ID、`primary_org_email` → Department Email、`primary_org_name` → Department という対応が不明 |
| 対応方針 | `user_to_rakumo_row()` 関数の実装方針（カラムマッピング表）をプログラム設計書に追加する |

### 【中程度】R2-05：GroupMembersテーブルのKey定義

| 項目 | 内容 |
|------|------|
| 対象ファイル | AppSheet設計書 2.3、Google Cloud設計書 5.4.3 |
| 問題 | AppSheet の GroupMembers テーブルは `id` をKeyとしているが、Cloud SQL DDL の `UNIQUE KEY uq_group_member (group_email, member_email)` と整合性の確認が必要。AppSheetでの重複レコード検知は `id` ではなく実質的な主キー（group_email + member_email）で行うべき |
| 対応方針 | AppSheet設計書 2.3 に「重複チェックはgroup_email + member_emailの複合で行う（AppSheet Virtual Column等で実現）」と補足を追記する |

### 【軽度】R2-06：ユーザーCSVエクスポートのカラム定義不明

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.2、Google Cloud設計書 10.1 |
| 問題 | `GET /api/users/export` の出力CSVカラム定義が記載されていない。rakumoプロファイルCSV（28列）と同じか、ユーザーマスタ全47項目を含むか不明 |
| 対応方針 | Google Cloud設計書 10.3 の API詳細（または要件定義書 9.3 ユーザーCSV仕様）に、エクスポートCSVのカラム定義（または「ユーザーマスタ全47項目＋変更種別」と明記）を追加する |

## 観点3：実現可能性

### 【重大】R3-01：Cloud Run タイムアウト3,600秒と数万件GWS API処理の乖離

| 項目 | 内容 |
|------|------|
| 対象ファイル | Google Cloud設計書 4.2、プログラム設計書 6.1 |
| 問題 | Google Workspace Admin SDK のユーザー作成APIは最大1 req/秒（事例による推定）のため、1万件処理に約2.7時間（9,700秒）が必要。Cloud Runの最大タイムアウト3,600秒（1時間）では処理しきれない |
| 対応方針 | 以下いずれかを採用し、設計書に明記する：①GWS APIバッチリクエスト（1回で複数操作）の活用 ②Cloud Run Jobsへの切り替え（タイムアウト制限の緩和） ③「一括処理は最大N件まで」という件数制限の設定（設計上の前提として明記） |

### 【中程度】R3-02：Google同期のポーリングと Cloud Run の同時実行制御

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.3、Google Cloud設計書 4.2 |
| 問題 | `wait_for_sync_completion` で最大10分（600秒）ポーリング中、Cloud RunのconcurrencyはデフォルトでHTTPリクエストが待機するため、concurrency=10のうちの1スレッドが占有される。Google同期が複数同時実行された場合の制御が不明 |
| 対応方針 | Google同期は1インスタンスのみ実行（`sync_logs` テーブルのrunningレコード確認で排他制御）する方針を、プログラム設計書の実装方針に明記する |

### 【中程度】R3-03：rakumo Profile API アップロード完了確認の実装方針

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.4、要件定義書 11.6 |
| 問題 | プログラム設計書ではStep4として「IMPORT LOCKが解除されるまでポーリング → 完了確認」と記載されているが、ポーリング間隔・最大待機時間・タイムアウト処理が明示されていない |
| 対応方針 | プログラム設計書 6.4 にポーリング実装コード（Google同期の `wait_for_sync_completion` に相当する関数）を追加する |

### 【軽度】R3-04：CSV署名付きURL の有効期限（1時間）とユーザー操作タイミング

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 6.6、AppSheet設計書 4.2 A-04〜06 |
| 問題 | 署名付きURL（有効期限1時間）をポップアップ表示後、ユーザーが1時間以内にダウンロードしなければURLが無効化される。AppSheet側でのURL有効期限の案内表示がない |
| 対応方針 | AppSheet設計書 4.2 の実行後メッセージに「ダウンロードリンクの有効期限は1時間です」と追記する |

### 【軽度】R3-05：HMAC-SHA1署名のhm.new誤記

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 5.3 |
| 問題 | コードスニペット内で `hmac.new(...)` と記載されているが、Pythonの正しい関数名は `hmac.new(...)` ではなく `hmac.new(key, msg, digestmod)` であり、この記法は正しいが `hmac.new` は Python 3では削除予定。正確には `hmac.new` → `hmac.HMAC(...)` または `hmac.new(key, msg, digestmod)` を確認する必要がある |
| 対応方針 | Python 3.12 では `hmac.new(key, msg, digestmod)` は `hmac.new` ではなく `hmac.HMAC` コンストラクタで実装する。コードスニペットを `hmac.HMAC(key, msg, hashlib.sha1).digest()` に修正する |

### 【軽度】R3-06：structlog ライブラリが requirements.txt 記載なし

| 項目 | 内容 |
|------|------|
| 対象ファイル | プログラム設計書 2.1（本番依存ライブラリ）、8.1（ロギング設計） |
| 問題 | ロギング設計で `structlog` を使用するコード例を記載しているが、requirements.txt の本番依存ライブラリ一覧に `structlog` が含まれていない |
| 対応方針 | プログラム設計書 2.1 の使用ライブラリ一覧に `structlog>=24.0` を追加する |

## 観点4：要件網羅性

### 【重大】R4-01：他システム連携CSVフォーマットの定義なし

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.7、Google Cloud設計書 10.1、プログラム設計書 6.6 |
| 問題 | 要件定義書 4.7 で「他システム連携用CSVを作成（フォーマット1種類）」と記載されているが、どの設計書にも具体的なカラム定義がない。プログラム設計書 6.6 では生成方法のみ記載されている |
| 対応方針 | 要件定義書 9章（CSV仕様）または新セクションに、他システム連携CSVの具体的なカラム定義（項目名・順序・データ型）を追加する。フォーマットはお客様との協議で確定次第記載 |

### 【中程度】R4-02：ユーザーCSVエクスポートのカラム定義が未定義

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.2・9.3 |
| 問題 | 要件定義書 9.3 のユーザーCSV仕様は「インポート用CSVの列定義」であり、エクスポート（ダウンロード）時のCSV形式と同一かどうかが不明。また役職・社員番号等の選択項目について「等」と記載され、全項目が明示されていない |
| 対応方針 | 要件定義書 9.3 に「エクスポートCSV形式（ダウンロード用）＝インポートCSV形式（変更種別列は手動ダウンロード時のみ表示）」と明記する |

### 【中程度】R4-03：同姓同名ユーザーの氏名ルールが設計書に未反映

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.5、プログラム設計書 6.4 |
| 問題 | 要件定義書に「同姓同名であってもGoogleアカウントの名前をそのまま使用する（識別子等を付加しない）」と記載されているが、プログラム設計書の rakumoコンタクト更新の実装方針にこのルールが明記されていない |
| 対応方針 | プログラム設計書 6.4 の実装方針に「氏名はGoogleアカウントの Family Name・Given Name をそのまま出力する（同姓同名の識別子付加は行わない）」を追記する |

### 【中程度】R4-04：GWS ユーザー削除時のアカウント処理方式が未定義

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.2、プログラム設計書 6.1 |
| 問題 | CSVの変更種別「削除」でユーザーを処理する際、Google WorkspaceアカウントをAPI経由で「完全削除（users.delete）」するのか「停止（users.update で suspended=true）」するのかが、どの設計書にも明記されていない |
| 対応方針 | プログラム設計書 6.1 の実装方針に削除方式（完全削除 vs 停止）を明記する。運用リスクの観点から「まず停止、一定期間後に完全削除」という2段階方式も検討すること |

### 【中程度】R4-05：AppSheet のアップロードCSV形式とインポート処理の整合

| 項目 | 内容 |
|------|------|
| 対象ファイル | AppSheet設計書 3.2（CSVアップロードフォーム）、要件定義書 9章 |
| 問題 | AppSheet設計書では「CSVファイル選択」という記述のみで、アップロードされるCSVの形式（エンコードUTF-8 BOM付き等）をフォーム上でユーザーにどう案内するかが記載されていない |
| 対応方針 | AppSheet設計書 3.2 のCSVアップロードフォームに「CSVフォーマット要件（文字コード：UTF-8 BOM付き / 1行目はヘッダー行必須等）をUIに表示する」旨を追記する |

### 【軽度】R4-06：4.8 定期実行・予約実行の「予約実行」仕様が設計に不足

| 項目 | 内容 |
|------|------|
| 対象ファイル | 要件定義書 4.8、Google Cloud設計書 9章、AppSheet設計書 |
| 問題 | 要件定義書 4.8 は「定期実行および予約実行」を要件としているが、設計書では「定期実行（Cloud Scheduler）」のみ設計されており、「任意日時への予約実行」機能の設計がない |
| 対応方針 | 予約実行の実現方式（Cloud Scheduler の cron 変更 vs AppSheet上のUI設計）を確認し、要件として対応するか明記する |

### 【軽度】R4-07：CSVインポートエラー行のユーザーへの通知方法が未設計

| 項目 | 内容 |
|------|------|
| 対象ファイル | Google Cloud設計書 10.3、AppSheet設計書 4.2 |
| 問題 | Google Cloud設計書 10.3 では「レスポンス：処理件数（成功/エラー件数）」と記載されているが、エラーが発生した行の詳細（行番号・メールアドレス・エラー理由）をユーザーに通知する方法が設計されていない |
| 対応方針 | AppSheet設計書 4.2 A-01〜03のレスポンス処理に「エラーがある場合はエラー件数と詳細をポップアップ表示するか、OperationLogsで確認する」旨を追記する |

### 【軽度】R4-08：ログ保持期間の記述統一

| 項目 | 内容 |
|------|------|
| 対象ファイル | Google Cloud設計書 7.2、AppSheetスプレッドシート設計書 3 |
| 問題 | Google Cloud設計書は「Cloud Loggingのログ保持期間：30日」、スプレッドシート設計書は「1ヶ月分を目安にクレンジング」と記載。Cloud SQLの操作ログ（operation_logs等）の保持期間が未定義 |
| 対応方針 | Cloud SQL ログテーブルの保持期間（例：3ヶ月）をGoogle Cloud設計書またはプログラム設計書に追記する |
