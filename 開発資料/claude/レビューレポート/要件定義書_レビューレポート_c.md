## 1. 冪等性の観点

同一操作を複数回実行しても同じ結果になるか（安全に再実行できるか）を確認した。

---

### [冪等-1] CSVインポートの再実行時の動作が未定義 ★重大

**該当箇所：** 4.2節・4.3節・9章（CSV仕様）

**問題内容：**
CSVの「変更種別」カラムに「追加」と記載されたレコードを2回インポートした場合の動作が定義されていない。
Google Workspace Admin SDK の `users.insert` は、既存メールアドレスに対して実行すると `409 Conflict` を返す。

```
1回目：「追加」→ GWSにユーザー作成 ✓
2回目：「追加」→ GWSが 409 Conflict → 処理失敗 or 無視？
```

**対応方針（案）：**
- 「追加」で既存ユーザーが存在する場合は「変更」として扱う（UPSERT）
- または CSV二重取込を防ぐために `csv_imports` テーブルでファイルハッシュを管理し、重複実行をエラーにする
- いずれかを要件として明記すること

---

### [冪等-2] 「削除」CSVの再実行時の扱いが未定義 ★中

**該当箇所：** 4.2節・4.3節

**問題内容：**
「変更種別：削除」のCSVを2回実行した場合、GWS側で既に削除済みのユーザー/グループに対して削除APIを呼び出すと、エラー（`404 Not Found`）が返る可能性がある。

**対応方針（案）：**
削除対象が既に存在しない場合は **成功（スキップ）** として扱うことを要件に明記する。

---

### [冪等-3] rakumo Profile APIのアップロードエラー後の再実行フローが不完全 ★中

**該当箇所：** 4.5節（3ステップの説明）

**問題内容：**
「アップロードURL（upload_url）は1回限り有効。エラー時は手順2から再実行」と記載されているが、**手順1のロック確認を挟む必要があるかが不明**。
また、手順3（アップロード）が失敗した時点でロックが残存している場合、手順2のURL再発行が `409 Conflict` になる可能性がある。

**対応方針（案）：**
エラー時の再実行フローを以下のように明示する。
```
エラー発生時：
① LOCK APIでロック状態確認
② ロック中 → UNLOCK APIでロック解除（手動/自動待機）
③ 手順1（ロック確認）から再実行
```

## 2. 整合性の観点

文書内の記述が相互に矛盾していないかを確認した。

### [整合-1] ユーザーCSVの項目数とユーザーマスタ仕様の大幅な乖離 ★重大

**該当箇所：** 9.3節（ユーザーCSV）vs 5.1節（ユーザーマスタ項目）

**問題内容：**

| | 9.3 ユーザーCSV | 5.1 ユーザーマスタ |
|-|----------------|-----------------|
| カラム数 | 6列 | 22項目 |
| 氏名 | 「名前」（1列） | 「姓」「名」（別カラム） |
| 組織 | 「部署」（1列） | 主務組織・兼務組織1〜4（別カラム） |
| 未対応項目 | ―― | 姓よみ・名よみ・会社名・会社名よみ・内線番号・携帯電話・兼務組織 etc. |

ユーザーCSV（6列）でインポートした場合、マスタ22項目のうち大半の項目がCSVから設定できない。

**対応方針（案）：**
以下のいずれかを選択・明記する。
- (A) CSVのカラムをマスタ全項目に対応するよう拡張する（22列以上）
- (B) 不足項目は別途（GWS API取得や別CSVで）補完することを明記する
- (C) CSVは「差分更新」用のため全項目でなくてもよいと明記する

---

### [整合-2] rakumoプロファイルCSV項目とユーザーマスタの不一致 ★重大

**該当箇所：** 11.5節（プロファイルCSVフォーマット）vs 5.1節・Google Cloud設計書

**問題内容：**
rakumoにアップロードするCSV（28列）には、ユーザーマスタ（5.1）およびCloud SQLの `users` テーブルに**存在しない項目**が含まれている。

| rakumoプロファイルCSVの項目 | ユーザーマスタ5.1 | Cloud SQL usersテーブル |
|--------------------------|----------------|----------------------|
| Birthday（生年月日） | 未定義 | カラムなし |
| Business Address（住所） | 未定義 | カラムなし |
| Business Fax（FAX） | 未定義 | カラムなし |
| E-mail 2 Address | 未定義 | カラムなし |
| E-mail 3 Address | 未定義 | カラムなし |
| Primary（プライマリフラグ） | 未定義 | カラムなし |
| Display（表示フラグ） | 未定義 | カラムなし |
| Calendar/Contacts/Workflow/Board/Expense/Attendance Enabled | 未定義 | カラムなし |

これらの項目を持つCSVを生成するためのデータソースが存在しない。

**対応方針（案）：**
- ユーザーマスタ仕様（5.1）にrakumoプロファイルCSVの全項目を追加する
- Cloud SQLの `users` テーブルに対応カラムを追加する
- または「固定値・デフォルト値を設定する項目」として明記する（例：Primary=1固定、Enabled系=空欄固定）

### [整合-3] ユーザーマスタに「組織名」カラムが不足 ★中

**該当箇所：** 5.1節 vs Google Cloud設計書 `users` テーブル

ユーザーマスタ5.1には「主務組織」（組織名）・「兼務組織1〜4」（組織名）が含まれているが、Cloud SQL `users` テーブルには対応するメールアドレスカラム（`primary_org_email`等）しか存在せず、**組織名（テキスト）カラムがない**。

**対応方針（案）：**
`primary_org_name`・`concurrent_org1_name`〜`concurrent_org4_name` カラムの追加を検討すること。または、組織名はグループテーブルの `group_name` から取得する設計とし、その旨を明記すること。

## 3. 実現可能性の観点

### [実現-1] rakumoプロファイルアップロードの完了確認がプログラムから不可能 ★重大

**該当箇所：** 4.5節 手順④・11.3節（API一覧）

**問題内容：**
4.5節の手順④「完了確認」は「管理画面の『APIログの確認』ページで確認」と記載されている。
しかし、**rakumo Profile APIには完了状態を取得するAPIエンドポイントが存在しない**。

| rakumo Profile APIの提供機能 | 完了確認 |
|---------------------------|---------|
| CSV DOWNLOAD | ✗ 完了確認ではない |
| CSV CREATE URL | ✗ URL発行のみ |
| CSV UPLOAD | ✗ 非同期で202返却 |
| CSV IMPORT LOCK | △ ロック解除を確認できるが完了とは別 |
| CSV IMPORT UNLOCK | ✗ 解除のみ |

**対応方針（案）：**
- Cloud Runでのアップロード完了確認は「LOCK APIでロックが解除されたことを確認する」をもって代用とする（ロック解除 = 処理完了）ことを明記する
- または、完了確認は「非同期で処理されるため、Cloud Runは受付完了のみ記録し、完全な完了確認は行わない」と割り切ることを明記する
- 管理画面での目視確認はあくまで運用担当者の確認行為として位置付ける

---

### [実現-2] Cloud RunからローカルファイルサーバーへのCSV直接書き込みは不可 ★中

**該当箇所：** 4.7節

**問題内容：**
「ファイルサーバーへCSVをダウンロード」と記載されているが、Cloud Run（GCP上のコンテナ）からローカルネットワーク上のファイルサーバーへ直接書き込むことは通常できない（ネットワーク的に到達不可）。
10.4節に「ローカルシステムからの実行を前提とする」と記載があるが、実行主体と手順が不明確。

**現実的な実現方法：**
```
Cloud Run → Cloud Storage（CSV生成・保存）
             ↓（署名付きURL）
担当者PC  → Cloud Storage（CSVダウンロード）
             ↓（手動）
担当者PC  → ファイルサーバー（格納）
```

**対応方針（案）：**
4.7節に「Cloud Runは Cloud Storage にCSVを生成し、担当者が署名付きURLまたはAppSheetのダウンロード機能を使用してCSVをダウンロードし、ファイルサーバーに手動格納する」と明記する。

### [実現-3] Google同期ポーリングのURL仕様が曖昧 ★中

**該当箇所：** 4.4節 手順3・12.5節 手順3

**問題内容：**
「`selfLink` URLをGETしてポーリング」と記載されているが、APIリストに存在するのは以下の2つのみ。
- `GET /api/google-sync/v1/jobs/current`（現在の状況）
- `GET /api/google-sync/v1/jobs/last`（最終結果）

`selfLink` の値が `/jobs/current` なのか `/jobs/{job_id}` のような固有URLなのかが不明確。`/jobs/{job_id}` の場合、APIリストに対応エンドポイントが存在しない。

**対応方針（案）：**
ポーリング実装は `GET /api/google-sync/v1/jobs/current` を使用することを明記する（`selfLink` のGETではなく）。または rakumo側に `selfLink` で取得できるエンドポイントの仕様を確認する。

### [網羅-1] 2.3の機能⑤「定期実行・予約実行」が4章に存在しない ★重大

**該当箇所：** 2.3節（解決策）vs 4章（機能要件）

**問題内容：**

| 2.3 解決策 | 4章の対応機能要件 |
|----------|----------------|
| ①ユーザーマスタ機能 | 5章（ユーザーマスタ仕様） |
| ②GWS一括登録・更新 | 4.2・4.3 |
| ③rakumoコンタクト更新 | 4.5 |
| ④他システムCSV作成・DL | 4.7 |
| **⑤定期実行・予約実行** | **記載なし** ← 欠落 |
| ⑥Google Cloud DB構築 | 5章・6章 |

「定期実行・予約実行を行う機能」は Cloud Scheduler によって実現されるが、**4章の機能要件に独立した記述がない**。

**対応方針（案）：**
4章に「4.8 定期実行・予約実行機能」を追加し、以下を記載する。
- Cloud Scheduler によるジョブスケジューリング
- 定期実行の対象処理（例：平日早朝のGWS同期、rakumoコンタクト更新）
- 実行時刻・頻度の要件
